The Flood: Netty's Uncapped HTTP/2 CONTINUATION Frames Leads to Denial of Service (DoS)

Vulnerability Research | March 24, 2026
CVSS 4.0 Score 8.7 (High)
CWE Identification CWE-770
CVSS Vector String CVSS:4.0/AV:N/AC:L/AT:N/PR:N/UI:N/VC:N/VI:N/VA:H/SC:N/SI:N/SA:N
CVE CVE-2026-33871
Affected Component Netty DefaultHttp2FrameReader

Summary

A significant security flaw was identified in Netty's DefaultHttp2FrameReader, affecting applications that utilize its HTTP/2 stack. The vulnerability arises from the parser's willingness to accept an unbounded sequence of CONTINUATION frames within a single header block. While Netty implements a byte-level constraint via maxHeaderListSizeGoAway, this mitigation is bypassed by 0-byte frames, which are technically valid under RFC 9113. A remote, unauthenticated attacker can exploit this architectural oversight to exhaust CPU resources and monopolize server-side threads with negligible bandwidth overhead.

Impact

  1. Asymmetric CPU Exhaustion: The server incurs a full parse-and-verify cycle for every frame, while the attacker only pays a 9-byte wire cost, leading to severe resource starvation.
  2. Protocol-Level Blocking: The HTTP/2 specification requires that no other frames be processed on a connection while a header fragment sequence is pending, allowing a single stream to freeze the entire multiplexed connection.
  3. Service Unavailability: By targeting the I/O thread pool across multiple concurrent connections, an attacker can trigger a total Denial-of-Service (DoS) for all users.

Attack Prerequisites

The vulnerability exists in the default configuration of Netty's HTTP/2 implementation. No specialized authentication, custom headers, or administrative misconfigurations are required for exploitation. The attack leverages 0-byte CONTINUATION frames, which are explicitly permitted under RFC 9113 Section 6.10, making them difficult to filter at the protocol layer without risking spec non-compliance.

Description

The root cause of this vulnerability is an architectural omission in the verifyContinuationFrame() method of Netty's DefaultHttp2FrameReader. In the HTTP/2 protocol, CONTINUATION frames are used to transport header fragments that exceed the maximum frame size. Netty's implementation validates stream association and state but fails to enforce any upper bound on the number of frames permitted in a single sequence.

// DefaultHttp2FrameReader.java
private void verifyContinuationFrame() throws Http2Exception {
    verifyAssociatedWithAStream();
    if (headersContinuation == null) {
        throw connectionError(PROTOCOL_ERROR, "...");
    }
    if (streamId != headersContinuation.getStreamId()) {
        throw connectionError(PROTOCOL_ERROR, "...");
    }
    // BUG: Missing frame count validation
}

To prevent memory exhaustion, Netty tracks the total byte size of accumulated headers using maxHeaderListSizeGoAway. However, this defense mechanism assumes that each frame contributes a measurable payload. By transmitting 0-byte CONTINUATION frames, an attacker ensures the byte-count remains static while the server continues to process an infinite number of discrete frame objects.

Mitigation Bypass: 0-Byte Payload Evasion

This "flood" of frames triggers an extreme imbalance in resource consumption. The server must perform a full state-machine transition and verification for every incoming fragment, effectively pinning a worker thread until the connection is closed or the server crashes.

Vulnerability Lifecycle: Connection Monopolization

Conclusion

This vulnerability underscores a critical class of HTTP/2 implementation flaws where specification compliance (accepting valid empty frames) conflicts with resource management. Netty users should prioritize upgrading to a patched release that enforces hard limits on frame counts within header sequences. Until an upgrade is feasible, implementing request-level timeouts and monitoring for excessive CPU usage on I/O threads are recommended as interim mitigation strategies.
