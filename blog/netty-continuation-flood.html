The Flood: Netty's Uncapped HTTP/2 CONTINUATION Frames Leads to Denial of Service (DoS)

Vulnerability Research | March 24, 2026
CVSS 4.0 Score 8.7 (High)
CWE Identification CWE-770
CVSS Vector String CVSS:4.0/AV:N/AC:L/AT:N/PR:N/UI:N/VC:N/VI:N/VA:H/SC:N/SI:N/SA:N
CVE CVE-2026-33871
Affected Component Netty DefaultHttp2FrameReader

Summary

A Denial-of-Service vulnerability was discovered in Netty's DefaultHttp2FrameReader. The reader accepts an unbounded number of HTTP/2 CONTINUATION frames per header block, with no frame count limit enforced. The only existing mitigation — a byte-level cap via maxHeaderListSizeGoAway — is completely bypassed by sending 0-byte CONTINUATION frames, which are valid per RFC 9113. An unauthenticated remote attacker can exploit this to monopolize server threads with minimal bandwidth, causing CPU exhaustion and connection-level Denial-of-Service against any default-configured Netty HTTP/2 server.

Impact

  1. CPU Exhaustion: Each CONTINUATION frame triggers full parse-and-verify logic server-side at ~389ns per frame, creating a significant CPU cost asymmetry relative to the 9-byte attacker wire cost.
  2. Connection Monopolization: The HTTP/2 protocol requires all other frames on a connection to be blocked while a CONTINUATION sequence is pending. A single open CONTINUATION chain freezes the entire connection.
  3. Thread Pool Starvation: Attacking multiple connections simultaneously exhausts the server's I/O thread pool, causing complete service unavailability.

Attack Prerequisites

None. This vulnerability affects the default Netty HTTP/2 server configuration. No authentication, special headers, or server misconfiguration is required. Any attacker capable of opening an HTTP/2 connection — over TLS (h2) or cleartext (h2c) — can trigger the issue.

The attack leverages 0-byte CONTINUATION frames, which are explicitly valid under RFC 9113 Section 6.10. The server cannot reject them at the protocol layer without violating spec compliance.

Description

In HTTP/2, a HEADERS frame may be followed by any number of CONTINUATION frames to carry the remainder of a compressed header block. Netty's DefaultHttp2FrameReader enforces no limit on how many such frames it will accept before the sequence is closed with an END_HEADERS flag. The verification method responsible for validating CONTINUATION frames, verifyContinuationFrame(), checks only stream association and state — never the frame count.

// DefaultHttp2FrameReader.java, lines 381–393
private void verifyContinuationFrame() throws Http2Exception {
    verifyAssociatedWithAStream();
    if (headersContinuation == null) {
        throw connectionError(PROTOCOL_ERROR, "...");
    }
    if (streamId != headersContinuation.getStreamId()) {
        throw connectionError(PROTOCOL_ERROR, "...");
    }
    // NO frame count limit — frames accepted indefinitely
}

Netty does implement a secondary safeguard: maxHeaderListSizeGoAway tracks accumulated header bytes and issues a GOAWAY once the threshold is exceeded. However, this protection assumes each frame contributes a nonzero byte payload. Sending 0-byte CONTINUATION frames renders it inert.

Byte Limit Bypass: Why 0-Byte Frames Evade maxHeaderListSizeGoAway

// HeadersBlockBuilder.addFragment() — lines 710–711
// When len = 0, this condition is NEVER true regardless of frame count
if (headersDecoder.configuration().maxHeaderListSizeGoAway() - len <
        headerBlock.readableBytes()) {
    headerSizeExceeded(); // 10240 - 0 < 0 → always FALSE
}

The result is a two-layer failure. The frame count check is absent entirely from verifyContinuationFrame(), and the only fallback — the byte accumulator — is neutralized by the attacker simply sending frames with no payload. Both defenses must fail simultaneously for exploitation, and both do.

Attack Flow: Connection Monopolization via 0-Byte CONTINUATION

Proof-of-Concept Results (Docker, maven:3.9-eclipse-temurin-11):

--- Test 1: 0-byte CONTINUATION frames ---
  [VULNERABLE] Accepted 50001 frames (HEADERS + 50000 CONTINUATION) with NO error!
  maxHeaderListSizeGoAway (10240 bytes) was NEVER triggered.
  Wire cost: 450009 bytes (439 KB) of attacker bandwidth.

--- Test 2: 1-byte CONTINUATION frames (for comparison) ---
  Error after 10241 frames: ClosedChannelException
  Byte limit triggered at ~10241 frames (expected ~10240)

--- Test 3: CPU cost measurement ---
  Frames processed: 100001
  Time: 39 ms
  Attacker bandwidth: 878 KB
  Server CPU per frame: ~389 ns

The asymmetry is stark: the attacker pays 9 bytes of wire cost per frame while the server pays a full parse-and-verify cycle. At 100,000 frames, the attacker sends under 900 KB while the server performs 100,000 discrete operations.

Fix Status:

A fix commit (c2ef97dce9, "Limit the number of Continuation frames per HTTP2 Headers") exists on the branch remotes/origin/limit_continuation_frame but was never merged to any release branch of Netty 4.1.x or 4.2. There are no release tags on the fix branch. The separately assigned CVE-2025-55163 (MadeYouReset) addresses a different HTTP/2 vulnerability — stream count bypass — and does not cover this CONTINUATION flood.

Conclusion

This vulnerability is a textbook case of an incomplete defense: a byte-accumulation guard that assumes all frames carry bytes, leaving it trivially bypassed by a spec-compliant edge case. The absence of a frame count limit in verifyContinuationFrame() is the root cause, and the byte guard's blind spot compounds it. The same vulnerability class has already affected Apache httpd (CVE-2024-27316), Tomcat (CVE-2024-24549), nghttp2 (CVE-2024-28182), and Envoy (CVE-2024-30255). Administrators running Netty HTTP/2 servers should monitor for the release of a patched version incorporating the unmerged fix commit, or apply manual frame count limiting at the handler level in the interim.
